## Git / GitHub: рабочий процесс для участников команды
### Клонирование репозитория
```bash
git clone https://github.com/thesecondpioneer/insurance-cross-sell.git
cd insurance-cross-sell
```
Либо можете воспользоваться любым удобным git-клиентом, например у самого git есть графический интерфейс, а ещё мне нравится TortoiseGit. \
Дальнейшее руководство будет по консольному инструменту, если решите использовать клиент, то прошу разобраться, как делать всё нижеперечисленное в нём.

### Переключение на ветку разработки
```bash
git checkout dev
```
### Создание ветки для своей задачи

Каждый участник создаёт отдельную ветку от dev.
Название ветки должно быть коротким и отражать суть задачи:
```bash
git checkout -b feature/ml-training
# или
git checkout -b fix/data-preprocessing
```
### Работа над своей задачей
- Вносите изменения только в нужные директории.
- Добавляйте комментарии и понятные имена переменным.
- Проверяйте, что код запускается и не ломает общий проект.

После изменений:
```bash
git add .
git commit -m "Добавлен модуль обучения модели (model_training.py)"
```
### Отправка изменений на GitHub
Перед пушем всегда обновляйтесь с основной ветки разработки:
```bash
git pull origin dev --rebase
git push origin feature/ml-training
```
После этого на GitHub появится предложение создать Pull Request (PR).
Выберите ветку назначения dev и добавьте краткое описание, что сделано.

### Ревью и объединение (merge)
Тут порядок действий такой: Я проверяю PR, после одобрения изменения вливаются (merge) в dev. Если реквест не одобрен - обсуждаем.\
Ветка main защищена - напрямую туда пушить нельзя, в нее я буду пушить готовые версии проекта, на защиту пойдёт как раз последняя с этой ветки.

### Синхронизация изменений

Перед началом новой задачи обновляйте свою ветку до актуального состояния:
```bash
git checkout dev
git pull origin dev
```
Если вы работаете в своей ветке и хотите подтянуть свежие изменения:
```bash
git merge dev
```

### Правила оформления коммитов

Формат сообщений коммитов (на русском или английском):
```
[тип]: краткое описание
```
Примеры:
```
feat: добавлен модуль предсказаний в backend
fix: исправлена ошибка в обработке данных
docs: обновлён README.md
refactor: оптимизация кода обучения модели
```

### Советы
- Делайте маленькие коммиты — проще откатить и ревьюить.
- Всегда перед началом работы делайте git pull origin dev.
- Если не уверены — создайте черновую ветку test/<имя> и попробуйте.
